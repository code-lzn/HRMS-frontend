# HRMS-组织架构管理-前端

## 需求背景

本模块来源于 HRMS（人资管理系统）产品规格说明书中第 3 部分——组织架构管理。当前公司内部组织架构依赖 Excel 维护，存在层级关系不清晰、部门人员统计滞后、职位与职级体系不规范等问题。本模块旨在建立统一的组织架构数字化管理前端界面，涵盖部门多级树形可视化、职位与序列职级体系配置、部门人员实时统计展示等核心能力，同时为员工档案管理、考勤管理、薪资管理、审批流转等下游模块提供组织架构基础数据的查看入口。

## 详细设计

### 前端迭代目标

本次前端开发聚焦于组织架构管理的两大功能模块：

1. **部门管理**：部门树形结构展示与操作（新增/编辑/删除）、层级深度可视化、部门编码管理、负责人设置、排序管理、部门人数实时统计展示
2. **职位管理**：职位列表展示与操作（新增/编辑/删除）、职位序列筛选、职级范围配置、所属部门关联、默认试用期配置

### 迭代具体描述

#### 部门管理（树形视图 + 详情面板）

##### UI&交互

UED 地址：参见 PRD 文档中的部门管理原型图

整体布局采用 **左右分栏** 结构：左侧为部门树（可伸缩），右侧为部门详情面板。

**部门树（左侧面板）交互：**

| **交互**         | **行为**                                             |
| ---------------- | ---------------------------------------------------- |
| 点击节点         | 选中节点（高亮），右侧展示该部门详情                 |
| 展开/折叠        | 点击箭头展开或折叠下级部门                           |
| 右键菜单（可选） | 快捷操作：新增子部门、编辑、删除                     |
| 搜索过滤         | 支持输入关键字,上级部门高亮匹配搜索                  |
| 节点文案         | 格式：`{部门名称} ({员工数})`，人数 > 0 时显示 Badge |

**部门详情（右侧面板）交互：**

| **区域**   | **内容**                                                        |
| ---------- | --------------------------------------------------------------- |
| 基本信息   | Descriptions 组件展示：名称、编码、上级部门、负责人、排序、描述 |
| 统计信息   | 部门人数（含子部门汇总）、子部门数量                            |
| 子部门列表 | 仅展示直接子部门简要卡片（名称+人数），点击可跳转到该子部门详情 |
| 操作区     | HR/管理员可见：编辑、删除、新增子部门；其他角色仅查看           |

**删除确认弹窗交互：**

| **场景** | **交互设计** |
| --- | --- |
| 正常删除（无子部门、无员工） | 普通 Popconfirm："确认删除该部门？" — 确认后执行 |
| 有子部门（错误码 30001） | Modal 展示子部门列表，提示"请先删除以下子部门" |
| 有在职员工（错误码 30002） | Modal 展示员工数量，提示"该部门下有 N 名在职员工，请先转移" |

###### 部门树查询接口

```plain
GET /api/v1/departments/tree
```

**请求参数**

| **参数** | **类型** | **必填** | **描述**                   |
| -------- | -------- | -------- | -------------------------- |
| —        | —        | —        | 无请求参数，返回完整部门树 |

**响应格式**

```json
{
  "code": 0,
  "message": "success",
  "data": [
    {
      "id": 1,
      "name": "总公司",
      "code": "00",
      "parentId": null,
      "managerId": 1000,
      "managerName": "王总",
      "sortOrder": 0,
      "employeeCount": 156,
      "description": "公司总部",
      "children": [
        {
          "id": 10,
          "name": "技术部",
          "code": "01",
          "parentId": 1,
          "managerId": 1001,
          "managerName": "张三",
          "sortOrder": 1,
          "employeeCount": 45,
          "description": "技术研发部门",
          "children": []
        }
      ]
    }
  ]
}
```

> **说明**：`employeeCount` 为该节点及其所有下属部门在职员工数之和。`children` 为空数组 `[]` 表示叶子节点。数据范围：HR 专员和系统管理员返回全量；部门主管仅返回本人管辖的部门子树；普通员工返回本人所在部门信息。`sortOrder` 控制同级部门排序，越小越靠前。

###### 部门详情查询接口

```plain
GET /api/v1/departments/{id}
```

**请求参数**

| **参数** | **类型** | **必填** | **描述**            |
| -------- | -------- | -------- | ------------------- |
| id       | Long     | 是       | 部门 ID（路径参数） |

**响应格式**

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 10,
    "name": "技术部",
    "code": "01",
    "parentId": 1,
    "parentName": "总公司",
    "managerId": 1001,
    "managerName": "张三",
    "sortOrder": 1,
    "level": 1,
    "employeeCount": 45,
    "childCount": 2,
    "children": [
      { "id": 101, "name": "前端组", "code": "0101", "employeeCount": 15 },
      { "id": 102, "name": "后端组", "code": "0102", "employeeCount": 20 }
    ],
    "description": "技术研发部门",
    "createTime": "2026-01-15T09:00:00",
    "updateTime": "2026-06-20T14:30:00"
  }
}
```

> **说明**：`children` 仅返回直接子部门的简要信息（id/name/code/employeeCount），不递归展开孙级。`parentName` 为上级部门名称，根部门时为 null。`level` 为动态计算的层级深度。

```json
  "code": 30001,
  "message": "该部门下存在子部门，请先删除子部门",

  }

```

```json
{
  "code": 30002,
  "message": "该部门下有在职员工，请先转移员工",
  "data": {
    "employeeCount": 45
  }
}
```

#### 部门新增/编辑（表单页）

##### UI&交互

UED ：参见 PRD 文档中的部门新增/编辑弹窗原型图

部门新增和编辑共用同一个弹窗表单（Modal + Form），通过 `mode` 区分 create / edit 模式。编辑模式下回填当前部门数据。新增根部门时上级部门为空，新增子部门时上级部门自动填充。

##### UI&交互

UED 地址：参见 PRD 文档中的职位新增/编辑弹窗原型图

职位新增和编辑共用同一个弹窗表单（Modal + Form），通过 `mode` 区分 create / edit 模式。编辑模式下回填当前职位数据。

**核心联动**：选择职位序列后，职级下限/上限的校验范围自动更新——M 序列 [1,5]、P 序列 [1,10]、S 序列 [1,5]，且职级下限必须 ≤ 职级上限。

## 其他

### 风险评估

| **风险项** | **影响** | **应对措施** |
| --- | --- | --- |
| 部门节点数量大（>1000）导致树渲染卡顿 | 用户体验 | 使用 Ant Design Tree 虚拟滚动；考虑分页加载子节点的懒加载模式替代全量加载 |
| 部门树人数统计实时性要求高 | 数据准确度 | 操作后立即刷新缓存；设置合理的缓存过期时间（30s）平衡实时性和性能 |
| 后端返回的嵌套 children 结构不一致 | 前端报错 | 数据转换时做好防御性处理，处理 null/undefined 场景 |
| 权限角色定义变更 | 按钮可见性逻辑需调整 | 将角色权限判断集中到统一的权限 hooks，避免分散在各组件中 |
| 浏览器兼容 | IE 不支持 | 明确仅支持 Chrome 90+ / Firefox 88+ / Edge 90+ / Safari 14+ |

### 稳定性保障

- **兼容性**：支持 Chrome 90+ / Firefox 88+ / Edge 90+ / Safari 14+
- **最低分辨率**：1366 × 768
- **数据刷新**：写操作（新增/编辑/删除）成功后立即刷新关联查询缓存，确保数据一致性
- **错误兜底**：所有 API 调用统一错误拦截，网络异常和业务错误码分别处理，避免白屏或未捕获异常
- **空状态处理**：部门树无数据时展示引导创建；职位列表无数据时区分"暂无数据"与"无搜索结果"
