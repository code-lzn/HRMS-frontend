# 2. 详细设计

## 2.1 前端迭代目标

建设统一的审批中心前端模块，包括：

1. **审批工作台**：待办列表（按业务类型筛选 + 委托路由）、已办列表、待办数量红点角标
2. **审批详情页**：业务信息 + 审批节点时间线 + 审批操作区（通过/拒绝/转交）
3. **委托审批管理**：设置委托（委托人+时间段）、取消委托、查询委托关系
4. **公共审批组件**：审批时间线组件（被入转调离/个人中心复用）、审批操作区组件
5. **与其他模块集成**：审批详情页根据 `bizType` 支持跳转到对应的业务详情页

**技术栈：** React + TypeScript + Umi + Ant Design + Zustand + umi-request + dayjs

**请求层封装：** 使用 UMI 内置的 `umi-request`（底层基于 axios），在 `app.ts` 的 `request` 配置中统一设置 baseURL、超时 30s、请求拦截器（自动注入 token）、响应拦截器（code≠0 Toast + 401 跳转登录）。待办数量通过 Zustand store 管理，请求成功后更新 store，全局导航栏订阅展示红点。

### 2.2.3 审批详情页

##### UI&交互

> UED 地址：待补充

这是审批中心最核心的页面，承载审批决策。

- **页面布局**：
  - 顶部：面包屑导航 + 审批标题 + 状态标签
  - 左侧（主区域）：业务信息卡片 + 审批时间线
  - 右侧（操作区）：审批操作按钮组（待审批节点时显示）
- **业务信息卡片**：展示该审批关联的业务摘要信息（根据 `bizType` 从后端获取）
  - 入职审批：姓名 / 部门 / 职位 / 预计入职日期
  - 转正审批：员工姓名 / 试用期 / 评价
  - 调岗审批：原部门→新部门 / 原职位→新职位
  - 离职审批：员工姓名 / 离职日期 / 离职类型
  - 请假审批：请假类型 / 时间段 / 天数
  - 补卡审批：补卡日期 / 原因
  - 薪资批次审批：月份 / 核算人数 / 总额
- **跳转链接**：业务信息卡片提供"查看业务详情"链接（根据 bizType 路由到对应模块详情页）
- **审批时间线**：复用入转调离模块的 `ApprovalTimeline` 组件
- **审批操作区**：当前用户为当前节点审批人时，展示操作按钮组
  - 通过按钮 + 拒绝按钮 + 转交按钮
  - 通过：弹出输入框（意见选填）→ 确认 → 调接口
  - 拒绝：弹出输入框（理由必填）→ 确认 → 调接口
  - 转交：弹出人员选择器 → 选择新审批人 → 调接口
- **撤回操作**：申请人为当前用户且当前为第一节点时，展示"撤回"按钮

##### 前端逻辑

###### 业务类型 → 业务详情页路由映射

| bizType          | 路由路径                                | 说明       |
| ---------------- | --------------------------------------- | ---------- |
| ONBOARDING       | /onboarding/detail/{bizId}              | 入职详情   |
| PROBATION        | /probation/detail/{bizId}               | 转正详情   |
| TRANSFER         | /transfer/detail/{bizId}                | 调岗详情   |
| RESIGNATION      | /resignation/detail/{bizId}             | 离职详情   |
| LEAVE            | /attendance/leave/detail/{bizId}        | 请假详情   |
| CARD_REPLENISH   | /attendance/replenish/detail/{bizId}    | 补卡详情   |
| SALARY_BATCH     | /salary/batch/detail/{bizId}            | 薪资批次   |

###### 审批操作按钮权限

| 按钮   | 显示条件                                           | 二次确认 |
| ------ | -------------------------------------------------- | -------- |
| 通过   | 当前用户 = 当前节点审批人（含委托）且节点状态=待审批 | Y        |
| 拒绝   | 同上                                               | Y        |
| 转交   | 同上                                               | N        |
| 撤回   | 当前用户 = 申请人 且 当前节点=第一节点 且 实例状态=审批中 | Y    |

###### 操作结果处理

- 通过/拒绝/转交成功后 → Toast 提示 → 如果是待办列表进入的，返回列表刷新；否则刷新详情
- 撤回成功后 → 返回列表刷新
- 失败 → Toast 显示后端错误信息（如"该节点已被处理"）

###### 设置委托审批 `POST /api/v1/approvals/delegates`

# 5. 其他

## 5.1 风险评估

| **风险项**                         | **风险等级** | **应对措施**                                                 |
| ---------------------------------- | ------------ | ------------------------------------------------------------ |
| 审批数据实时性（多人并发审批）     | 中           | 审批操作后立即重新请求列表接口，确保数据最新     |
| 委托路由数据不一致                 | 低           | 待办列表由后端统一处理委托路由，前端不自行判断               |
| 审批操作后页面状态不同步           | 中           | 审批操作成功后刷新详情+列表，使用 queryClient.invalidateQueries |
| 业务类型扩展（新增审批类型）       | 低           | bizType 路由映射使用配置化方式，新增类型只需增加配置项       |
| 逾期审批用户体验                   | 低           | 逾期标记红色背景 + "已逾期X小时"提示，排序靠前               |

## 5.2 稳定性保障

- **审批幂等性**：审批操作按钮点击后立即 disabled + loading，接口返回前不可重复点击
- **并发冲突**：后端返回"该节点已被处理"(40004)时，前端自动刷新页面获取最新状态
- **委托时间校验**：开始时间不早于当前、结束时间晚于开始时间、时间段不重叠
- **路由跳转**：bizType → 业务详情路由映射使用常量化配置，避免硬编码
- **兼容性**：Chrome 90+ / Firefox 88+ / Edge 90+ / Safari 14+
