# 2. 详细设计

## 2.1 前端迭代目标

本次迭代实现入转调离四大异动场景的前端页面，包括：

1. **入职管理**：入职列表页、入职申请表单页（新建/编辑草稿）、入职详情页（含审批进度时间线）、确认入职/放弃入职操作
2. **转正管理**：转正列表页、转正申请表单页、转正详情页、待转正员工提醒列表
3. **调岗管理**：调岗列表页、调岗申请表单页、调岗详情页、员工调岗历史查看
4. **离职管理**：离职列表页、离职申请表单页、离职详情页
5. **公共审批组件**：审批进度时间线组件、审批操作组件（通过/拒绝/转交），与审批中心复用

**技术栈：** React + TypeScript + Umi + Ant Design + AntV + Zustand + umi-request + dayjs

**请求层封装：** 使用 UMI 内置的 `umi-request`（底层基于 axios），在 `app.ts` 的 `request` 配置中统一设置 baseURL、超时 30s、请求拦截器（自动注入 token）、响应拦截器（code≠0 统一 Toast + 401 跳转登录）、错误处理等。各业务模块通过自定义 Hook（如 `useOnboardingList`）封装请求逻辑，返回 `{ data, loading, error, refresh }` 给页面组件使用。

## 5.1 风险评估

| **风险项**                     | **风险等级** | **应对措施**                                               |
| ------------------------------ | ------------ | ---------------------------------------------------------- |
| 审批进度数据与审批中心不一致   | 中           | 统一使用审批中心接口获取审批数据，前端不做缓存             |
| 表单联动逻辑复杂（入职表单）   | 低           | 使用 Form.useWatch 或状态管理，分步校验                     |
| 状态流转前端校验不完整         | 中           | 前后端双重校验，前端仅做 UI 约束，最终以后端返回为准       |
| 调岗/离职员工状态实时性        | 低           | 操作后调接口刷新列表，请求成功后更新 Zustand store，列表组件订阅 store 自动更新 |
| 大量列表数据性能               | 低           | 分页 + 虚拟列表（如超过 500 条），请求数据存入 Zustand，跨组件复用 |

## 5.2 稳定性保障

- **容量评估**：单页面并发约 50-100 人，列表分页默认 20 条/页
- **兼容性**：Chrome 90+ / Firefox 88+ / Edge 90+ / Safari 14+，分辨率 1366×768+
- **错误处理**：统一错误拦截，网络异常 Toast 提示 + 重试按钮
- **Loading 态**：表格 Skeleton、表单 Submit Button loading、详情 Spin
- **Empty 态**：空列表、空搜索结果友好提示
- **回滚方案**：前端静态资源版本化，异常时回滚到上一版本 CDN
