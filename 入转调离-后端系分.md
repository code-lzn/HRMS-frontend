## 1. 项目背景

> 对本次项目的背景以及目标进行描述，方便开发者理解需求，对齐上下文。

### 1.1 业务背景

本模块来源于 HRMS（人资管理系统）产品规格说明书中第 5 部分——入转调离流程。当前公司内部员工入离职依赖纸质流程和线下沟通，存在流程不透明、审批周期长、数据同步不及时、历史追溯困难等问题。本模块旨在实现入转调离全流程线上化，覆盖入职、转正、调岗、离职四大异动场景，打通审批流，自动同步员工档案状态变更，为考勤、薪资等下游模块提供准确的员工状态数据。

## 3. 功能模块

> 描述本次迭代涉及的功能与场景。

### 3.1 核心功能

1. **入职管理**：候选人信息录入 → 草稿保存 → 提交审批 → 审批流转（部门负责人 → HR负责人） → 审批通过后自动生成工号、创建账号、写入员工档案、发送欢迎邮件
2. **转正管理**：系统每天扫描试用期到期员工（**入职日期 + 试用期（月）- 7 天**）并提醒 → HR发起转正评估 → 转正审批（部门负责人 → HR负责人） → 通过后状态变更为正式 / 拒绝后延长试用或辞退
3. **调岗管理**：HR发起调岗申请 → 原部门负责人审批，知情确认→ 新部门负责人审批，接收确认 → HR负责人备案，最终确认 → 调岗生效（更新部门/职位/职级/汇报人，工号不变，记录调岗历史）
4. **离职管理**：HR发起离职申请 → 部门负责人确认交接 → HR负责人审批，通过 → 离职生效（状态变更为待离职→已离职，账号禁用，从考勤组移除，薪资结算至离职日）

    │       │                                    │

    │       ├── 拒绝 → 状态=已拒绝，通知HR         │

    │                                            │

    │       │                                    │
    │       ├── 通过 → 进入后处理                 │
    │       ├── 拒绝 → 状态=已拒绝                │
    │       └── 转交 → 变更审批人                 │

    │                                            │

    │    （由审批中心统一处理，详见审批中心系分）   │
    └────────────────────┬─────────────────────┘

    ┌──────────────────────────────────────────┐
    │           审批通过后处理（事务）            │
    │                                            │
    │  1. 生成工号（年份+部门编码+序号）          │

    ┌──────────────────────────────────────────┐

    │  - 员工状态=试用期                         │
    │  - 开始计算考勤                            │

    └──────────────────────────────────────────┘

### 5.5 通用审批流驱动流程

```
发起申请（创建ApprovalInstance + ApprovalNode）
     │
     ▼
当前节点审批人待办
     │
     ├── 通过
     │     │
     │     ├── 是否最后一个节点？
     │     │     ├── 是 → 审批通过 → 回调业务Service → ...
     │     │     └── 否 → 流转到下一节点（下一审批人待办）
     │     │
     ├── 拒绝
     │     │
     │     └── 审批结束 → 状态=已拒绝 → 通知申请人
     │
     ├── 转交
     │     │
     │     └── 变更当前节点审批人 → 通知新审批人
     │
     └── 撤回（仅限第一节点，HR本人操作）
           │
           └── 审批实例→已撤回，业务申请回退为草稿（可再次编辑提交，对应PRD 5.1.2）
```

#### 9.2.6 提交入职审批

```
POST /api/v1/onboarding/{id}/submit
```

**前置条件：** 仅 status=草稿(1) 时可提交

**请求参数：** 无（入职申请完整字段校验已在创建/更新时完成）

**响应格式：**

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 101,
    "status": 2,
    "statusDesc": "审批中",
    "approvalInstanceId": 2001,
    "approvalNodes": [
      {"nodeName": "部门负责人审批", "nodeOrder": 1, "approverId": 100, "approverName": "李四", "status": 1, "statusDesc": "待审批"},
      {"nodeName": "HR负责人审批", "nodeOrder": 2, "approverId": 200, "approverName": "王五", "status": 1, "statusDesc": "待审批"}
    ]
  }
}
```

**后端处理逻辑：**
1. 校验所有必填字段（12个业务字段）
2. 状态变更：草稿→审批中
3. 调用 ApprovalFlowService.createInstance() 创建审批实例
4. 构建审批节点：Node1=部门负责人，Node2=HR负责人

## 10. 关键技术设计

> 描述入转调离模块涉及的核心技术方案与难点。

### 10.1 与审批中心的集成

**使用方：** 入职管理（入职审批流转 + 审批通过后处理回调）、转正管理（转正审批流转 + 转正结果回调）、调岗管理（调岗审批流转 + 调岗生效回调）、离职管理（离职审批流转 + 离职生效回调）

入转调离的四大流程均依赖审批中心进行审批流转。本模块通过实现 `ApprovalCallback` 接口与审批中心集成：

- **ApprovalFlowService**（审批中心提供）：负责审批实例创建、节点流转
- **ApprovalCallback**（本模块实现）：审批通过后执行入职后处理/转正生效/调岗生效/离职生效

各业务的审批节点链由审批中心的 `ApprovalNodeBuilder` 按业务类型动态构建。完整审批流引擎设计、委托审批机制、审批超时处理见 [《审批中心-后端系分》](./审批中心-后端系分.md) 第 10 节。

### 10.2 入职后处理事务设计

**使用方：** 入职管理 — 审批全部通过后，在回调中执行工号生成 + 员工档案写入（employee/personal_info/work_info），需保证事务一致性

**问题**：入职审批通过后需同时完成工号生成、档案写入等多个操作，任何一步失败需全部回滚（登录账号创建由登录鉴权模块负责）。

**方案**：`@Transactional` 保证事务一致性

```
@Transactional(rollbackFor = Exception.class)
public void postApprovalProcess(Long onboardingId) {
    // 1. 查询入职申请
    OnboardingApplication app = onboardingMapper.selectById(onboardingId);

    // 2. 生成工号（行级锁防并发）
    String employeeNo = employeeNoGenerator.generate(app.getDepartmentId());

    // 3. 创建员工档案
    Employee employee = buildEmployee(app, employeeNo);
    employeeMapper.insert(employee);

    // 4. 写入档案信息
    //    personal_info：name/gender/phone/email/idCard 来自申请表；address（现居住地址）初始化为空，员工入职后可在个人中心自行编辑（对应PRD 4.1.2）
    //    work_info：departmentId/positionId/hireType/directReportId 来自申请表
    //    薪资与合同信息（salary_info）由HR在薪资管理模块单独设置（对应PRD 7.2员工薪资设置）
    personalInfoMapper.insert(buildPersonalInfo(app, employee.getId()));
    workInfoMapper.insert(buildWorkInfo(app, employee.getId()));

    // 5. 更新入职申请关联
    onboardingMapper.updateEmployeeId(app.getId(), employee.getId());
    onboardingMapper.updateStatus(app.getId(), OnboardingStatus.APPROVED_PENDING_JOIN);

    // 6. 发送MQ消息（事务提交后消费者才可见，使用RabbitMQ的事务消息或本地消息表）
    // 注：MQ发送在事务提交后执行，避免事务回滚时消息已发送
    TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() {
        @Override
        public void afterCommit() {
            // 登录鉴权模块进行账号创建（account=app.phone，密码等由鉴权模块负责）
            notificationProducer.sendOnboardingNotification(employee, app);
        }
    });
}
```

### 10.3 离职生效定时任务

**使用方：** 离职管理 — 离职审批通过后，员工状态变为"待离职"，定时任务在离职日期当天自动执行离职生效（状态变更、账号禁用、考勤组移除、薪资结算）

**问题**：离职审批通过后员工状态变为"待离职"，需在离职日期当天自动变为"已离职"并执行相关清理。

**方案**：Spring `@Scheduled` 定时任务 + 分布式锁（Redis）防止多实例重复执行

```
@Scheduled(cron = "0 0 1 * * ?")  // 每天凌晨1点执行
public void executeResignationJob() {
    String lockKey = "resignation:job:lock";
    String lockValue = UUID.randomUUID().toString();

    try {
        // 获取分布式锁（过期时间10分钟）
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, lockValue, 10, TimeUnit.MINUTES);
        if (!locked) return;

        // 查询今天到期的待离职员工
        List<ResignationApplication> list = resignationMapper
            .selectByStatusAndDate(ResignationStatus.APPROVED, LocalDate.now());

        for (ResignationApplication app : list) {
            try {
                resignationService.executeResignation(app.getId());
            } catch (Exception e) {
                log.error("离职生效处理失败: resignationId={}", app.getId(), e);
                // 单条失败不影响其他记录
            }
        }
    } finally {
        // 释放锁（仅释放自己的锁）
        String currentValue = redisTemplate.opsForValue().get(lockKey);
        if (lockValue.equals(currentValue)) {
            redisTemplate.delete(lockKey);
        }
    }
}
```

**离职生效处理（事务）**：

```
@Transactional
public void executeResignation(Long resignationId) {
    ResignationApplication app = resignationMapper.selectById(resignationId);

    // 1. 员工状态 → 已离职
    employeeMapper.updateStatus(app.getEmployeeId(), EmployeeStatus.RESIGNED);

    // 2. 从考勤组移除
    attendanceGroupMemberMapper.deleteByEmployeeId(app.getEmployeeId());

    // 3. 更新离职申请状态
    resignationMapper.updateStatus(resignationId, ResignationStatus.RESIGNED);
    resignationMapper.updateActualDate(resignationId, LocalDate.now());

    // 4. 记录操作日志
    auditLogService.log("RESIGNATION_EXECUTED", app.getEmployeeId(), "离职生效");

    // 5. MQ通知（事务提交后）
    registerAfterCommit(() -> {
        // 通知鉴权模块禁用登录账号（is_deleted=1由鉴权模块处理）
        notificationProducer.sendResignationNotification(app);
    });
}
```

### 10.5 调岗/离职的员工状态校验

**使用方：** 调岗管理（创建调岗申请时校验员工在职状态）、离职管理（创建离职申请时校验员工在职状态）

**调岗条件**：员工在职状态 = 试用期(1) 或 正式(2)

**离职条件**：员工在职状态 = 试用期(1) 或 正式(2)

**校验实现**：

```
public void validateEmployeeStatus(Long employeeId, String operation) {
    Employee employee = employeeMapper.selectById(employeeId);
    if (employee == null) {
        throw new BusinessException("员工不存在");
    }

    EmployeeStatus status = employee.getStatus();
    switch (operation) {
        case "TRANSFER":
        case "RESIGNATION":
            if (status != EmployeeStatus.PROBATION && status != EmployeeStatus.REGULAR) {
                throw new BusinessException("当前员工状态不允许" +
                    ("TRANSFER".equals(operation) ? "调岗" : "离职"));
            }
            break;
        case "PROBATION":
            if (status != EmployeeStatus.PROBATION) {
                throw new BusinessException("仅试用期员工可发起转正");
            }
            break;
    }
}
```
