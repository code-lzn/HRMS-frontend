## 1. 项目背景

### 1.1 业务背景

本模块来源于 HRMS（人资管理系统）产品规格说明书中第 8 部分——审批中心。入转调离、请假、补卡、薪资批次等业务均产生审批需求，当前各业务独立处理审批逻辑导致审批规则分散、待办入口不统一、委托审批难以实现。本模块旨在建设统一的审批中心，提供通用的审批流引擎、集中的待办/已办工作台、灵活的委托审批机制，为各业务模块提供审批能力支撑。

### 1.2 审批类型汇总（PRD 8.1）

| **业务类型** | **申请者** | **审批流**                              |
| ------------ | ---------- | --------------------------------------- |
| 入职审批     | HR         | 部门负责人 → [HR负责人]                 |
| 转正审批     | HR         | 部门负责人 → HR负责人                   |
| 调岗审批     | HR         | 原部门负责人 → 新部门负责人 → HR负责人   |
| 离职审批     | HR         | 部门负责人 → HR负责人                   |
| 请假审批     | 员工       | 按请假类型+天数动态路由（见PRD 6.3.4）  |
| 补卡审批     | 员工       | 直接上级                                |
| 薪资批次审批 | HR         | 财务专员 → [老板]                       |

## 3. 功能模块

### 3.1 核心功能

1. **通用审批流引擎**：提供审批实例创建、节点流转（通过/拒绝/转交）、撤回、回调等通用能力，各业务模块接入时只需实现回调接口
2. **审批人工作台**：统一的待办列表（按业务类型筛选）、已办列表、审批详情（含业务信息+审批历史时间线）
3. **委托审批**：审批人可设置委托人在指定时间段内代为审批，委托期间审批任务自动路由给被委托人，被委托人操作时系统记录代理标识（"XXX 代 YYY 审批"），original_approver_id 保留原审批人信息用于审计追溯
4. **审批超时处理**：每级审批 48 小时超时，超时自动升级并催办

### 8.2 表结构设计

#### 8.2.1 审批实例表 approval_instance

```sql
CREATE TABLE IF NOT EXISTS `approval_instance` (
    `id`                  BIGINT UNSIGNED   NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `biz_type`            VARCHAR(32)       NOT NULL COMMENT '业务类型：ONBOARDING/PROBATION/TRANSFER/RESIGNATION/LEAVE/CARD_REPLENISH/SALARY_BATCH',
    `biz_id`              BIGINT UNSIGNED   NOT NULL COMMENT '业务主键ID（关联各业务申请表ID）',
    `title`               VARCHAR(128)      NOT NULL COMMENT '审批标题，如"张三的入职申请"',
    `status`              TINYINT           NOT NULL DEFAULT 1 COMMENT '审批状态：1=审批中 2=已通过 3=已拒绝 4=已撤回',
    `applicant_id`        BIGINT UNSIGNED   NOT NULL COMMENT '申请人ID',
    `current_node_order`  INT               NOT NULL DEFAULT 1 COMMENT '当前审批节点序号',
    `create_time`         DATETIME          NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time`         DATETIME          NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_biz` (`biz_type`, `biz_id`),
    KEY `idx_applicant_id` (`applicant_id`),
    KEY `idx_status` (`status`),
    KEY `idx_create_time` (`create_time`)
) DEFAULT CHARACTER SET = utf8mb4 COMMENT = '审批实例表';
```

**设计说明：**
- `biz_type + biz_id` 唯一索引确保每个业务记录只有一个审批实例
- `biz_type` 用 VARCHAR 而非 ENUM，便于后续增加审批类型不需改表结构
- `current_node_order` 记录当前审批到第几个节点，避免遍历所有节点找当前活跃节点

#### 8.2.2 审批节点表 approval_node

```sql
CREATE TABLE IF NOT EXISTS `approval_node` (
    `id`                    BIGINT UNSIGNED   NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `instance_id`           BIGINT UNSIGNED   NOT NULL COMMENT '审批实例ID',
    `node_name`             VARCHAR(64)       NOT NULL COMMENT '节点名称，如"部门负责人审批"、"HR负责人审批"',
    `node_order`            INT               NOT NULL COMMENT '节点顺序号（从1开始）',
    `approver_id`           BIGINT UNSIGNED   NOT NULL COMMENT '当前审批人ID',
    `original_approver_id`  BIGINT UNSIGNED            COMMENT '原审批人ID（转交/委托场景记录，用于审计"XXX代YYY审批"）',
    `status`                TINYINT           NOT NULL DEFAULT 1 COMMENT '节点状态：1=待审批 2=已通过 3=已拒绝 4=已转交',
    `comment`               VARCHAR(512)               COMMENT '审批意见',
    `operate_time`          DATETIME                   COMMENT '操作时间',
    `create_time`           DATETIME          NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time`           DATETIME          NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (`id`),
    KEY `idx_instance_id` (`instance_id`),
    KEY `idx_approver_id_status` (`approver_id`, `status`),
    KEY `idx_instance_id_order` (`instance_id`, `node_order`)
) DEFAULT CHARACTER SET = utf8mb4 COMMENT = '审批节点表';
```

**设计说明：**
- `idx_approver_id_status` 联合索引覆盖待办列表的高频查询（WHERE approver_id = ? AND status = 1）
- `original_approver_id` 记录最初审批人，转交时写入，审计时可追溯"谁最初应该审批"

#### 8.2.3 委托审批表 approval_delegate

```sql
CREATE TABLE IF NOT EXISTS `approval_delegate` (
    `id`                BIGINT UNSIGNED   NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `delegator_id`      BIGINT UNSIGNED   NOT NULL COMMENT '委托人ID（原审批人）',
    `delegate_id`       BIGINT UNSIGNED   NOT NULL COMMENT '被委托人ID（代理审批人）',
    `start_time`        DATETIME          NOT NULL COMMENT '委托开始时间',
    `end_time`          DATETIME          NOT NULL COMMENT '委托结束时间',
    `enabled`           TINYINT           NOT NULL DEFAULT 1 COMMENT '是否启用：0=已取消 1=生效中',
    `create_time`       DATETIME          NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time`       DATETIME          NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (`id`),
    KEY `idx_delegator_id` (`delegator_id`),
    KEY `idx_delegate_id` (`delegate_id`),
    KEY `idx_time_enabled` (`start_time`, `end_time`, `enabled`)
) DEFAULT CHARACTER SET = utf8mb4 COMMENT = '委托审批表';
```

**设计说明：**
- `idx_time_enabled` 覆盖委托路由的核心查询（WHERE now() BETWEEN start_time AND end_time AND enabled = 1）
- 委托取消时不物理删除，设 enabled=0 保留审计记录

### 8.3 索引设计说明

| **表名**           | **索引**                        | **用途**                                       |
| ------------------ | ------------------------------- | ---------------------------------------------- |
| approval_instance  | uk_biz (biz_type, biz_id)       | 唯一约束：每个业务记录仅一个审批实例           |
| approval_instance  | idx_status                      | 按状态筛选审批列表                             |
| approval_node      | idx_approver_id_status          | 待办列表核心查询（审批人+待审批状态）          |
| approval_node      | idx_instance_id_order           | 按实例查所有节点+排序                          |
| approval_delegate  | idx_time_enabled                | 委托路由核心查询（当前时间是否在有效委托内）   |

## 10. 关键技术设计

### 10.1 审批流引擎：策略模式构建节点链

**使用方：** 全部审批业务类型 — 入职审批（OnboardingNodeBuilder）、转正审批（ProbationNodeBuilder）、调岗审批（TransferNodeBuilder）、离职审批（ResignationNodeBuilder）、请假审批（LeaveNodeBuilder）、补卡审批、薪资批次审批

审批流引擎的核心挑战是不同业务类型需要不同的审批节点链。用策略模式解决：

```
ApprovalNodeBuilder 接口
  ├── OnboardingNodeBuilder    → Node1:部门负责人, Node2:HR负责人
  ├── ProbationNodeBuilder     → Node1:部门负责人, Node2:HR负责人
  ├── TransferNodeBuilder      → Node1:原部门负责人, Node2:新部门负责人, Node3:HR负责人
  ├── ResignationNodeBuilder   → Node1:部门负责人, Node2:HR负责人
  ├── LeaveNodeBuilder         → 按请假类型+天数动态路由
  └── ...
```

`ApprovalFlowService.createInstance()` 内部根据 bizType 查表（或 Spring 容器）获取对应的 Builder，调用 `builder.build()` 生成节点链。

加新审批类型只需新增一个 Builder 实现类，注册到工厂中即可，审批引擎代码零改动。

### 10.2 审批人确定机制

**使用方：** 审批节点构建器（OnboardingNodeBuilder/ProbationNodeBuilder 等）— 创建审批实例时按节点角色查询对应审批人；部门负责人节点（各业务审批）、HR负责人节点（入职/转正/调岗/离职审批）、直接上级节点（请假/补卡审批）、指定审批人（转交场景）

| **节点角色**   | **确定方式**                                     | **说明**                   |
| -------------- | ------------------------------------------------ | -------------------------- |
| 部门负责人     | `SELECT manager_id FROM department WHERE id = ?`  | 直接查部门表               |
| HR 负责人      | 查询系统配置中角色=HR负责人的用户列表             | 可能多个人，任一审批即可   |
| 直接上级       | `SELECT direct_report_id FROM employee_work_info` | 请假/补卡审批用            |
| 指定审批人     | 业务申请时显式传入                               | 转交场景                   |

### 10.3 委托审批的路由机制

**使用方：** 审批工作台 — 查询待办列表（`getPendingList`）时调用 `resolveApprover()` 进行委托路由，将本应发给委托人的审批任务自动转发给被委托人；被委托人操作时系统记录代理标识，`original_approver_id` 保留原审批人信息用于审计追溯

审批任务路由的核心逻辑在 `ApprovalDelegateService.resolveApprover()`：

```
输入：原始审批人 ID
输出：实际审批人 ID（可能是本人，也可能是被委托人）

逻辑：
1. 查询 approval_delegate WHERE delegator_id = ? AND enabled = 1
   AND NOW() BETWEEN start_time AND end_time
2. 有 → 返回 delegate_id
3. 无 → 返回原始审批人 ID（自己）
```

查询待办列表时，需要同时查两部分：
- `WHERE approver_id = currentUserId`（我自己的待办）
- `WHERE approver_id IN (SELECT delegate_id FROM approval_delegate WHERE delegator_id = currentUserId AND ...)`（别人委托给我的）

两部分 UNION 后返回。

### 10.4 审批超时处理

**使用方：** 全部审批节点 — 每个审批节点创建时均发送一条 48 小时延迟消息，覆盖入职/转正/调岗/离职/请假/补卡/薪资批次等所有业务类型的审批节点

**方案**：RabbitMQ 延迟队列

审批节点创建时发送延迟消息：
```
approval_node 创建
  → MQ: send(message={instanceId, nodeId}, delay=48h)
  → 48h 后消费者收到消息
    → 查节点状态：
      - 仍为"待审批" → 自动升级（通知上级审批人）+ 催办通知
      - 已处理 → 忽略
```

**为什么用延迟队列而不是定时任务轮询？**
- 轮询需要扫描全表，审批实例多了之后性能差
- 延迟队列天然精确到每个节点，不需要全表扫描

### 10.5 与业务模块的集成方式

**使用方：** 入转调离模块（OnboardingServiceImpl/ProbationServiceImpl/TransferServiceImpl/ResignationServiceImpl 实现 ApprovalCallback）、考勤管理模块（请假/补卡审批回调）、薪资管理模块（薪资批次审批回调）

业务模块接入审批中心只需要两步：

**步骤一**：实现 `ApprovalCallback` 接口

```java
@Service
public class OnboardingServiceImpl implements ApprovalCallback {
    @Override
    public void onApproved(String bizType, Long bizId) {
        // 入职后处理：生成工号、创建账号、写入档案、发邮件
        postApprovalProcess(bizId);
    }

    @Override
    public void onRejected(String bizType, Long bizId) {
        // 更新入职申请状态=已拒绝
        onboardingMapper.updateStatus(bizId, OnboardingStatus.REJECTED);
    }
}
```

**步骤二**：提交审批时调用审批引擎

```java
// 业务 Service 中：
approvalFlowService.createInstance(
    ApprovalBizType.ONBOARDING,  // 业务类型
    application.getId(),          // 业务主键
    application.getApplicantId()  // 申请人
);
```

审批引擎不关心业务细节，只负责流转；业务模块不关心审批流转，只负责审批通过后的处理。

### 10.6 缓存策略

**使用方：** 审批工作台（待办数量红点 — 顶部导航角标实时更新）、委托审批模块（委托审批关系 — 路由查询加速）、审批流引擎（审批节点配置 — 节点构建时读取缓存避免每次查DB）

| **缓存内容**         | **位置** | **过期时间** | **更新策略**                   |
| -------------------- | -------- | ------------ | ------------------------------ |
| 待办数量（红点）     | Redis    | 实时         | 新审批+1，审批完成-1           |
| 委托审批关系         | Redis    | 委托结束时间 | 委托设置/取消时更新            |
| 审批节点配置         | Redis    | 1 小时       | 配置变更时主动失效             |

### 10.7 消息队列设计

**使用方：** 审批流引擎（审批任务通知 — 新节点激活时通知审批人；审批结果通知 — 通过/拒绝后通知申请人）、审批超时处理（延迟消息 — 48h 超时自动触发；催办通知 — 超时后催办审批人）、审计服务（审批操作审计 — 通过/拒绝/转交/撤回等操作记录）

| **消息类型**     | **交换机**          | **路由键**            | **消费者** | **触发时机**               |
| ---------------- | ------------------- | --------------------- | ---------- | -------------------------- |
| 审批任务通知     | hrms.notification   | approval.task         | 通知服务   | 新审批节点激活             |
| 审批结果通知     | hrms.notification   | approval.result       | 通知服务   | 审批通过或拒绝             |
| 审批超时延迟     | hrms.delay          | approval.timeout      | 审批服务   | 审批节点创建时（48h延迟）  |
| 催办通知         | hrms.notification   | approval.remind       | 通知服务   | 超时后催办                 |
| 审批操作审计     | hrms.audit          | approval.audit        | 审计服务   | 通过/拒绝/转交/撤回       |

## 13. 风险评估

| **风险项**               | **等级** | **影响范围** | **应对措施**                                         |
| ------------------------ | -------- | ------------ | ---------------------------------------------------- |

| 审批人离职/调岗           | 低       | 审批流转     | 审批人变更时自动触发转交，通知管理员手动处理         |
